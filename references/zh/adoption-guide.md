# 采用指南

本指南用于把 `create-jarvis-skill` 带入真实公司环境，并从抽象兴趣推进到可用的 试点。

## 采用的目标

目标不是用一个生成出来的仓库打动别人。

目标是创造第一份可见证据，证明：
- agents 能找到正确的公司上下文，
- agents 能参与真实交付工作，
- 组织也能保留下它们学到的东西。

## 推荐的采用顺序

### 1. 从一条痛点明显的闭环开始

选择一条满足以下条件的闭环：
- 经常发生，
- 横跨不止一个工作面，
- 并且痛感足够强，让人能明显感受到改进。

好的例子包括：
- 重复发生的 bug triage
- 跨 frontend / backend / QA / docs 的功能交付
- 基于真实工程工作生成 release note

不要一开始就抱着“把整个公司都画出来”的宏大目标。

### 2. 选择狭窄但真实的范围

对于 试点，请定义：
- 少量核心 sources，
- 少量核心 repos，
- 一条真实 workflow，
- 以及当前负责这项工作的相关人员。

这样可以让 试点 扎根于真实执行，而不是停留在架构表演层面。

### 3. 只构建 试点 真正需要的部分

生成：
- build brief，
- inventories，
- 第一版 backlog，
- ownership map，
- rollout plan，
- 以及只有最高杠杆的 skill 骨架。

### 4. 先完成适配，再宣称成功

把占位符替换成：
- 真实 owners，
- 真实 repos，
- 真实 systems，
- 真实 access paths，
- 以及真实 workflow boundaries。

如果 试点 读起来仍然像模板，它就还没准备好。

### 5. 证明这条闭环

用这个 试点 完成真实工作。

重点寻找这样的证据：试点 降低了
- 重新发现信息的成本，
- 上下文交接成本，
- 跨 repo 等待成本。

### 6. 把经验写回去

试点 结束后，更新：
- JARVIS 材料，
- skill backlog，
- ownership map，
- 以及如果 rollout 暴露出更好模式时，对方法本身也进行更新。

## 在早期采用中应当预期什么

早期 rollout 往往并不均衡。

常见现实包括：
- 有些系统容易接入，另一些则受政治或组织因素影响
- repo-local guidance 通常缺失或不一致
- ownership 往往要等 试点 强迫澄清后才会清晰
- 人们通常在第一条闭环真正跑通之后才会理解其价值

这很正常。

## 一个简单的采用测试

当第二个人或第二个 agent 能利用生成出来的产物继续推进这条闭环，而不必从零开始时，说明 试点 已经在发挥作用。
