# Rollout 与 Ownership

把 JARVIS 当作一个多 owner rollout，而不是一次性建设任务。

## 为什么 ownership 很重要

JARVIS 通常横跨：
- 不同数字资产，
- 不同 repos，
- 不同 teams，
- 不同审批边界，
- 以及不同类型的 权威来源。

一个 agent 或一个人类可以启动这项工作，但想要 durable rollout，就必须有明确 ownership。

## 建议的 rollout 阶段

### Stage 0 — Clarify
- 定义业务意图
- 选择第一条高价值闭环
- 识别初始 owners

### Stage 1 — Pilot
- 只盘点第一条闭环所需的 sources、repos 和 workflows
- 创建第一版 backlog
- 生成第一批 scaffolds

### Stage 2 — Operationalize
- 为 试点 区域补上缺失的 source / repo / workflow skills
- 建立 回写 规则
- 让 owners 之间的 handoff 可靠运行

### Stage 3 — Scale
- 把覆盖范围扩展到相邻 workflows
- 改进治理与质量检查
- 降低团队间的重新发现成本

## 需要记录的 owner 类型

至少要识别：
- business owner
- knowledge owner
- repo owner(s)
- workflow owner(s)
- JARVIS maintainer / steward

## 良好 rollout 的输出

一个好的 rollout 应当让人容易回答：
- 接下来由谁继续，
- 他们下一步该建设什么，
- 什么被卡住了，
- 哪些部分已经可用，
- 当前 JARVIS 到达了哪个成熟度层级。

## 需要避免的失败模式

不要把早期 scaffolding 说得像组织已经拥有完整 JARVIS 覆盖一样。
