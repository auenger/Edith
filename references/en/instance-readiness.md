# Instance Readiness

Use this checklist to judge whether a generated JARVIS is just a scaffold, a pilot, or a truly usable instance.

## Level 0 — Scaffold

A scaffold exists when:
- root structure exists,
- maintenance guidance exists,
- and some templates or inventories exist.

A scaffold is **not yet operational**.

## Level 1 — Pilot

A pilot exists when:
- one real business loop is defined,
- key sources, repos, and workflows for that loop are inventoried,
- at least some source / repo / workflow skills are drafted,
- and another agent or owner can continue without starting from zero.

A pilot proves direction, not completeness.

## Level 2 — Operational instance

An operational instance exists when:
- modules contain real knowledge, not placeholders,
- source routing is real and usable,
- repo-local truth is linked correctly,
- workflow closure is explicit,
- and START → WORK → END writeback is happening in practice.

At this level, the JARVIS can already reduce rediscovery and handoff cost in day-to-day work.

## Level 3 — Mature instance

A mature instance exists when:
- historical knowledge is deep and structured,
- repeated failures are captured as patterns,
- decisions and rejected ideas are recorded with rationale,
- cross-cutting knowledge is actively maintained,
- ownership is stable,
- and the system keeps improving from actual use.

At this level, JARVIS is no longer just a repository. It is part of how the organization thinks and works.

## Readiness checks

Before calling an instance “operational” or “mature”, confirm:
- [ ] business modules have real overview / issue / decision content
- [ ] sources have real routing docs
- [ ] cross-cutting knowledge exists where needed
- [ ] company-specific JARVIS skill entry exists
- [ ] repo skills and workflow skills exist where the main loops require them
- [ ] writeback happens after real work, not only during setup
- [ ] another owner or agent can continue without rediscovering fundamentals
