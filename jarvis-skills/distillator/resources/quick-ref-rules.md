# 速查卡压缩规则

速查卡目标压缩至源文的 ~5%。这不是摘要，是极端信号提取，只保留操作必需信息。

## 提取（保留）

- 构建 / 测试 / 运行 / Lint 命令（精确命令，可直接复制粘贴）
- 硬约束（违反即故障的规则，Top 5）
- 易错点（按频率和严重度排序，Top 5）
- 接口清单（Method + Path + 一句话用途，无 request/response schema）
- 数据模型名称 + 用途 + 2-3 关键字段（无完整字段列表）
- 关键外部依赖和集成点

## 丢弃（即使事实正确也去除）

- 完整接口 request/response schema
- 完整数据模型定义（所有字段、索引、约束）
- 设计决策及理由
- 被拒绝的方案及原因
- 环境搭建步骤
- 数据库迁移历史
- 所有 TODO/FIXME
- 配置细节（构建命令之外）
- 代码示例
- 架构设计说明
- 业务领域背景知识
- 测试策略描述
- 依赖版本号（除非有兼容性约束）

## 格式规则

- 验证命令必须可直接复制粘贴执行，不加解释
- 约束每条一行，祈使语气："禁止 X"、"必须 Y"
- 易错点格式："症状 — 原因"
- 接口表格只用三列：Method / Path / Purpose
- 数据模型每模型一行，不超过 3 个关键字段
- 不使用散文段落
- 不使用装饰性格式
- 总输出严格控制在 2000 token 以内

## 与 --jarvis-mode 组合

当 `--quick-ref` 与 `--jarvis-mode=repo-skill` 组合时：
- 优先保留接口清单和验证命令
- 约束偏向代码层面（不可违反的编码规则）
- 易错点偏向运维层面（部署和运行时陷阱）

当 `--quick-ref` 单独使用时：
- 使用通用速查卡压缩规则
- 均衡分配各部分内容
