# 语义拆分策略

当源内容超过 15,000 token 或 token_budget 要求时，按语义一致性拆分而非按大小切割。

## 为什么语义拆分优于大小拆分

按大小切割会破坏连贯性。下游加载"第 2/4 部分"得到的是碎片。语义拆分产生自包含的主题集群，可以按需选择性加载——"给我技术决策部分"——更有用且更省 token。

## 拆分流程

### 1. 识别自然边界

完成初始提取和去重后，寻找自然语义边界：
- 不同的问题域或功能区域
- 不同的利益相关者视角（用户、技术、业务）
- 时间边界（当前状态 vs 未来规划）
- 范围边界（包含/排除/延期）
- 阶段边界（分析、设计、实现）

选择下游工作流可能独立加载的边界。

### 2. 分配信息到分片

每条信息分配到最相关的分片。跨分片的放根文件。

跨主题项（涉及多个分片的内容）：
- 影响所有区域的约束 → 根文件
- 广泛影响的决策 → 根文件
- 分片特定的决策 → 分片文件

### 3. 生成根文件（_index.md）

包含：
- **导向**（3-5 条）：蒸馏了什么、从哪些源、给谁用、几部分
- **分片清单**：每个分片文件名 + 一行描述
- **跨主题项**：跨多个分片的事实、决策和约束
- **范围摘要**：高层级的包含/排除/延期

### 4. 生成分片文件

每个分片必须自包含——只加载一个分片的读者应该能理解它。

每个分片包含：
- **上下文头**（1 行）：本分片覆盖 [主题]。第 N/M 部分。
- **分片内容**：按主题组织的密集信息点，遵循压缩规则
- **交叉引用**（如需要）：指向其他分片的相关内容

### 5. 输出结构

```
{名称}-distillate/
├── _index.md           # 根文件：导向、跨主题项、分片清单
├── 01-{主题slug}.md    # 自包含分片
├── 02-{主题slug}.md
└── 03-{主题slug}.md
```

## 大小目标

指定 token_budget 时：
- 根文件：约 20% 预算（导向 + 跨主题项）
- 剩余预算按内容密度按比例分配到各分片
- 某分片超出比例份额 → 更激进压缩或子拆分

未指定 token_budget 但需要拆分时：
- 每个分片目标 3000-5000 token
- 根文件尽可能小但保持独立可用
